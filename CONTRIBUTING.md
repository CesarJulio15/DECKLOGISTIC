## Diretrizes de Contribuição

Este documento estabelece normas, expectativas e procedimentos formais para contribuições ao projeto, garantindo consistência, qualidade técnica e colaboração responsável.

## 1. Objetivo
Estas diretrizes servem para:
- Definir expectativas claras.
- Minimizar retrabalho e ruído operacional.
- Manter qualidade do código e documentação.
- Promover colaboração técnica e profissional.

## 2. Abrindo Issues
Antes de abrir uma issue:
1. Verifique se já existe uma issue relacionada.
2. Descreva o problema de forma objetiva:
   - Contexto
   - Comportamento atual
   - Comportamento esperado
   - Passos para reproduzir
   - Evidências (logs, prints, exemplos mínimos)

Para novas features:
- Explique o caso de uso.
- Justifique o impacto.
- Proponha abordagens viáveis.

## 3. Enviando Pull Requests
Para aumentar a chance de aprovação:
1. Abra uma issue antes, salvo correções triviais.
2. Use um branch próprio para cada alteração.
3. Inclua no PR:
   - Descrição objetiva da mudança
   - Justificativa técnica
   - Evidências de testes ou validações
   - Referência à issue relacionada
4. Mantenha o escopo enxuto.
5. Siga o estilo de código do projeto.
6. Inclua testes quando aplicável.

PRs fora do padrão, com escopo exagerado ou sem justificativa técnica podem ser recusados.

## 4. Código de Conduta
O projeto adota o Contributor Covenant v2.0.
Comportamento profissional é obrigatório.
Violação resulta em advertência, suspensão ou banimento conforme diretrizes oficiais.

## 5. Comunicação
Use os canais adequados:
- Issues: dúvidas e problemas.
- Pull Requests: contribuições diretas.
- Discussions (se habilitado): debates amplos.

Evite mensagens privadas para temas que precisam ser rastreados.

## 6. Estrutura de Pastas
Caso o projeto siga esta estrutura:
- /src → código-fonte
- /docs → documentação
- /tests → testes
- /tools → scripts auxiliares

Explique novas estruturas no PR.

## 7. Padrões de Commit
Use Conventional Commits:
feat: nova funcionalidade
fix: correção de bug
docs: atualização de documentação
refactor: melhoria interna sem alteração funcional
test: testes
chore: manutenção interna

Commits genéricos serão recusados.

## 8. Processo de Revisão
PRs serão avaliados quanto a:
- Qualidade técnica
- Clareza
- Impacto no projeto
- Riscos
- Aderência às diretrizes

Ajustes podem ser solicitados. PRs inativos podem ser fechados.

## 9. Licença
Ao contribuir, você concorda que sua contribuição será distribuída sob a mesma licença do projeto.

## 10. Agradecimento
A evolução do projeto depende de contribuições bem estruturadas. Obrigado pelo seu comprometimento técnico.
